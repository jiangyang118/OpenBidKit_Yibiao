# 历史任务与过程档案存储改造计划

## 1. 背景

当前系统更像“当前工作区”的处理模式：用户上传一份文件后，后续解析、生成、导出结果主要围绕当前流程状态展开。这个模式适合单次操作，但不适合长期使用：

- 用户重新上传新文件时，容易覆盖或清空旧流程状态。
- “重置”容易被理解为只清空当前任务，但实际可能清掉当前模块下的过程文件。
- 用户无法在一个列表里看到每次上传、每次生成、每一步产物之间的对应关系。
- 后续如果做网页版和多人协作，必须提前把“用户、项目、任务、文件、产物”拆清楚。

本计划的目标是把系统从“单一当前流程”改造成“可追溯的任务档案系统”：每次上传和每次生成都形成一条独立历史记录，所有输入文件、过程文件、AI 输出、导出文件都能回看、恢复和继续处理。

## 2. 核心原则

1. 数据库保存索引、状态和关系，文件系统保存大文件和生成结果。
2. 每次上传默认创建一个新的任务记录，不再覆盖旧任务。
3. 每一步生成都保存快照，既能看到最终结果，也能回看过程。
4. 重置只影响当前任务，不删除历史任务；删除历史任务需要用户显式操作。
5. 先支持本地 Electron 版本，再为网页版多人账号做平滑迁移。
6. 原有功能先保持兼容，避免一次性大改导致现有流程不可用。

## 3. 推荐信息结构

整体关系建议为：

```text
用户 / 本地工作区
  -> 项目 Project
    -> 任务 Run
      -> 输入文件 Files
      -> 步骤快照 Step Snapshots
      -> 过程产物 Artifacts
      -> 导出文件 Exports
      -> 操作事件 Events
```

在本地版本中，可以先没有真实账号，但数据结构上预留 `owner_id`、`project_id`、`run_id`，后续改网页版时只需要把本地用户替换成真实登录用户。

## 4. 数据库设计

建议新增一组通用工作流档案表，先服务“技术方案”和“完整标书”，后续扩展到商务标、废标分析、图片知识库等模块。

### 4.1 workflow_runs

记录一次完整任务。一次上传、一次导入、一次从旧任务复制继续，都可以是一条 run。

关键字段：

- `run_id`：任务唯一 ID。
- `project_id`：所属项目，可为空；后续接项目管理。
- `owner_id`：所属用户，本地版可先固定为 `local`。
- `module`：来源模块，例如 `technical_plan`、`bid_document`。
- `title`：用户可读标题，默认可用文件名 + 时间生成。
- `status`：`draft`、`running`、`completed`、`failed`、`archived`。
- `active_step`：当前停留步骤。
- `source_file_id`：主输入文件。
- `created_at`、`updated_at`、`archived_at`。
- `metadata_json`：扩展信息，例如招标项目名、采购人、模板 ID。

### 4.2 workflow_files

记录用户上传或导入的原始文件，以及解析后的中间文件。

关键字段：

- `file_id`：文件唯一 ID。
- `run_id`：所属任务。
- `role`：`source`、`attachment`、`parsed_markdown`、`reference`。
- `original_name`：原始文件名。
- `mime_type`、`file_ext`、`file_size`。
- `sha256`：内容哈希，用于去重和校验。
- `storage_path`：相对 `workspace/` 的存储路径。
- `parser_name`、`parser_version`。
- `created_at`。

### 4.3 workflow_step_snapshots

记录每一步的输入、输出和状态。它解决“每一步生成过程没有地方看”的问题。

关键字段：

- `snapshot_id`：快照唯一 ID。
- `run_id`：所属任务。
- `step_key`：步骤标识，例如 `import_tender`、`extract_requirements`、`generate_outline`、`generate_content`、`export_word`。
- `status`：`pending`、`running`、`completed`、`failed`。
- `input_snapshot_json`：本步骤开始时使用的关键输入。
- `output_snapshot_json`：本步骤完成后的结构化结果摘要。
- `artifact_ids_json`：本步骤产生的产物 ID 列表。
- `task_id`：如果后台任务系统有任务 ID，在这里关联。
- `started_at`、`finished_at`。
- `error_json`：失败原因。

### 4.4 workflow_artifacts

记录过程产物，例如解析 Markdown、提取清单、目录 JSON、章节正文、AI 原始响应、校验报告等。

关键字段：

- `artifact_id`：产物唯一 ID。
- `run_id`：所属任务。
- `step_key`：来源步骤。
- `kind`：`markdown`、`json`、`docx`、`xlsx`、`log`、`ai_response`、`validation_report`。
- `title`：用户可读名称。
- `storage_path`：文件路径。
- `content_hash`：内容哈希。
- `metadata_json`：扩展信息。
- `created_at`。

### 4.5 workflow_exports

记录用户最终导出的文件。

关键字段：

- `export_id`：导出记录唯一 ID。
- `run_id`：所属任务。
- `export_type`：`word`、`markdown`、`excel`、`zip`。
- `title`：导出名称。
- `storage_path`：导出文件路径。
- `created_at`。

### 4.6 workflow_run_events

记录用户操作和系统事件，便于排查问题。

关键字段：

- `event_id`：事件唯一 ID。
- `run_id`：所属任务。
- `event_type`：`created`、`uploaded`、`step_started`、`step_completed`、`reset_current`、`exported`、`archived`。
- `message`：简短中文说明。
- `payload_json`：结构化详情。
- `created_at`。

## 5. 文件目录设计

SQLite 负责关系和索引，大文件放到 `userData/workspace/` 下的独立目录：

```text
workspace/
  runs/
    <run_id>/
      input/
        原始招标文件.docx
      parsed/
        tender.md
        tender-meta.json
      steps/
        step01-import/
          snapshot.json
          result.md
        step02-extract/
          requirements.json
          ai-response.md
        step03-outline/
          outline.json
        step04-content/
          content.json
          sections/
      artifacts/
        validation-report.json
        build-log.md
      exports/
        技术方案.docx
        缺口报告.xlsx
      logs/
        events.ndjson
```

路径写入数据库时建议保存相对路径，例如 `runs/<run_id>/input/source.docx`，避免用户目录变化后绝对路径失效。

## 6. 页面与交互设计

### 6.1 新增历史任务列表

在相关模块入口增加“历史任务”列表，至少展示：

- 创建时间。
- 任务标题。
- 原始文件名。
- 当前状态。
- 当前步骤。
- 最近导出结果。
- 所属模块。

列表操作：

- 打开任务：进入该任务的当前步骤和数据。
- 继续处理：把该任务恢复为当前编辑任务。
- 复制为新任务：保留输入和部分结果，另开一次新处理。
- 归档：从默认列表隐藏，但不删除文件。
- 删除：二次确认后删除数据库记录和文件目录。
- 导出任务包：打包输入、过程文件和结果，方便迁移或备份。

### 6.2 任务详情页

任务详情页展示一条清晰时间线：

```text
上传文件 -> 解析招标文件 -> 提取要求 -> 生成目录 -> 生成正文 -> 校验 -> 导出 Word
```

每一步可查看：

- 使用了哪些输入。
- 生成了哪些中间文件。
- AI 原始输出或结构化结果。
- 是否失败，失败原因是什么。
- 可下载的过程文件。

### 6.3 重置语义调整

需要把“重置”拆成不同含义：

- 重置当前步骤：只清空当前步骤和后续步骤的结果。
- 重置当前任务：清空当前 run 的流程结果，但保留原始上传文件。
- 新建任务：创建新 run，旧 run 保留在历史列表。
- 删除任务：显式删除历史记录和文件。

界面文案必须避免让用户误以为“重置”会删除全部历史。

## 7. 与现有功能的衔接

### 7.1 技术方案流程

当前技术方案已有 `technical_plan.json` 和相关 workspace 缓存。第一阶段不要强行删除旧结构，可以这样兼容：

1. 新上传招标文件时创建 `workflow_runs`。
2. 原始文件复制到 `workspace/runs/<run_id>/input/`。
3. 解析出的 Markdown 写入 `workspace/runs/<run_id>/parsed/tender.md`。
4. 原有 `technical_plan.json` 继续作为“当前打开任务”的兼容缓存。
5. 每完成一步，把关键结果同步保存到 `workflow_step_snapshots` 和 `workflow_artifacts`。

这样可以先不破坏现有页面，再逐步让页面直接读取 run 数据。

### 7.2 完整标书流程

完整标书已经更接近项目配置和构建日志模式，适合较早接入：

- 每次导入项目配置或新建标书项目时创建 run。
- 项目配置 JSON 作为 artifact 保存。
- 附件映射和材料包作为 file/artifact 保存。
- 缺口报告、校验报告、Word 成品进入 exports。

### 7.3 项目工作区

现有项目工作区可以继续作为上层组织方式。建议关系为：

```text
Project Workspace
  -> 多个 workflow_runs
```

同一个项目下可以有多次上传、多版方案、多次导出，避免用户为了保存历史而反复复制项目目录。

## 8. 分阶段实施计划

### Phase 1：建立通用档案底座

目标：先让系统具备保存历史记录的能力。

任务：

- 新增 SQLite 表：`workflow_runs`、`workflow_files`、`workflow_step_snapshots`、`workflow_artifacts`、`workflow_exports`、`workflow_run_events`。
- 新增 Main 侧服务：`workflowArchiveService.cjs`。
- 新增 IPC：创建任务、查询任务列表、读取任务详情、归档任务、删除任务。
- 新增文件目录工具：按 `run_id` 创建 `workspace/runs/<run_id>/`。
- 增加最小历史任务列表页面或弹窗。

验收：

- 创建新任务后数据库有记录。
- 文件能写入独立 run 目录。
- 历史列表能显示任务。
- 归档不删除文件。
- 删除任务会二次确认并清理对应目录。

### Phase 2：技术方案上传接入历史任务

目标：新上传文件不再覆盖旧文件，旧任务能在列表中看到。

任务：

- 技术方案上传时自动创建 run。
- 原始招标文件复制到 run 的 `input/`。
- 解析 Markdown 保存到 run 的 `parsed/`。
- 当前任务 ID 写入当前工作区状态。
- 新建任务按钮替代部分“重置后再上传”的操作。

验收：

- 上传 A 文件后生成 run A。
- 再上传 B 文件后生成 run B。
- run A 和 run B 都在历史列表中。
- 打开 run A 时能看到 A 的原始文件和解析结果。
- 上传 B 不删除 A 的文件。

### Phase 3：保存每一步生成过程

目标：每一步 AI 生成和处理结果都可追溯。

任务：

- Step01 导入展示保存快照。
- Step02 要求提取保存输入、输出、AI 响应和结构化 JSON。
- Step03 目录生成保存目录 JSON 和 Markdown。
- Step04 正文生成保存章节正文、每章生成状态和失败信息。
- 校验结果保存为 artifact。
- 失败步骤保存 `error_json`。

验收：

- 用户能看到每一步何时开始、何时结束。
- 每一步能查看对应产物。
- AI 生成失败时，历史详情能看到失败原因。
- 重新生成某一步不会覆盖旧快照，可以形成新版本。

### Phase 4：导出与结果归档

目标：最终 Word、Excel、Markdown、Zip 等导出结果都归属到任务。

任务：

- Word 导出写入 `workflow_exports`。
- 缺口报告、校验报告写入 `workflow_artifacts` 或 `workflow_exports`。
- 任务详情页展示最近导出。
- 支持导出任务包，包含输入、过程和结果。

验收：

- 同一任务多次导出都能在列表中看到。
- 用户能下载历史导出文件。
- 导出任务包后可以完整备份该任务。

### Phase 5：扩展到其他模块

目标：让商务标、废标分析、图片知识库等也能复用同一套历史任务能力。

任务：

- 梳理各模块的输入、步骤、产物和导出。
- 给每个模块定义 `module` 和 `step_key`。
- 统一历史任务列表筛选。
- 支持按项目、模块、时间、状态搜索。

验收：

- 不同模块的任务可以共存在同一历史列表中。
- 用户可以按模块过滤。
- 不同模块不会互相覆盖文件。

### Phase 6：为网页版和多人账号预留

目标：从本地单机平滑升级到 Web 多人协作。

任务：

- 把 `owner_id` 替换成真实用户 ID。
- 增加 `tenant_id` 或组织 ID。
- 文件存储从本地目录替换为对象存储，例如 S3、R2、OSS。
- SQLite 替换为 PostgreSQL 或 MySQL。
- 增加权限模型：创建者、协作者、只读查看者。
- 增加审计日志和文件下载权限校验。

验收：

- 用户只能看到自己或组织授权的任务。
- 同一项目可多人协作。
- 文件下载需要鉴权。
- 后端可以按用户、项目、任务查询完整链路。

## 9. 迁移策略

不要直接删除旧数据。建议迁移时采用保守策略：

1. 如果当前 workspace 已有技术方案数据，则创建一个 `legacy` run。
2. 把现有 `technical_plan.json`、`technical-plan/tender.md` 等文件登记为 legacy run 的 artifacts。
3. 保留原文件路径，同时新文件写入 `runs/<run_id>/`。
4. 页面默认打开最近一次 run。
5. 提供“导入旧工作区为历史任务”的一次性工具。

## 10. 风险与处理

### 10.1 存储空间增长

风险：每次生成都保存过程文件，长期使用后 workspace 会变大。

处理：

- 历史列表显示任务占用空间。
- 支持归档和删除。
- 后续支持只保留最终结果、清理 AI 原始响应等策略。

### 10.2 文件路径失效

风险：用户迁移电脑或修改目录后，绝对路径失效。

处理：

- 数据库只保存 workspace 相对路径。
- 导出任务包使用相对目录结构。

### 10.3 后台任务与当前任务错乱

风险：用户切换任务时，后台生成仍在写旧状态。

处理：

- 后台任务必须绑定 `run_id`。
- 写入前校验任务 ID。
- UI 显示“当前正在为哪个任务生成”。

### 10.4 旧页面依赖单例状态

风险：现有页面可能默认只读写一个当前状态。

处理：

- 第一阶段保留 current cache。
- 新档案系统先做旁路记录。
- 第二阶段逐步把页面数据源切到 run。

### 10.5 敏感文件安全

风险：招投标文件可能包含敏感信息。

处理：

- 本地版先不上传云端。
- Web 版必须做用户鉴权、下载鉴权、操作审计。
- 删除任务时明确提示会删除本地文件。

## 11. 验收清单

最小可用版本应满足：

- 用户上传新文件时，旧文件和旧结果不会被删除。
- 历史任务列表能看到每次上传记录。
- 每条任务能看到输入文件、当前状态和最近结果。
- 打开旧任务后能查看旧解析结果。
- 每一步生成的主要产物有记录。
- 重置当前任务不会删除其他任务。
- 导出的 Word 或报告能回到对应任务下查看。
- `cd client && npm run build` 通过。

## 12. 暂定开放问题

后续实现前需要确认：

1. 历史任务列表是放在每个模块内部，还是做一个全局“任务中心”。
2. 默认一个上传文件是否一定创建新任务，还是允许用户选择覆盖当前任务。
3. 旧任务默认保留多久，是否需要自动清理策略。
4. AI 原始响应是否全部保存，还是只保存结构化结果和最终文件。
5. 本地版是否需要先做“任务包导入/导出”，方便以后迁移到网页版。
6. Web 版第一阶段是个人账号模式，还是一开始就支持组织/团队空间。

## 13. 推荐先做的最小闭环

建议优先做一个小而完整的闭环：

1. 新增档案表和 run 目录。
2. 技术方案上传时创建 run。
3. 历史列表展示 run。
4. 原始文件和解析 Markdown 保存到 run。
5. 打开历史任务能看到原始文件名和解析结果。
6. 新上传文件不影响旧任务。

这个闭环完成后，用户就能立刻感受到“不会丢旧数据”，再继续补每一步快照、导出归档和多人协作。

## 14. Codex 执行入口

面向 Codex 的可执行任务清单见：`client/doc/历史任务过程归档TODO.md`。

后续实现时优先读取 TODO 文档，并按其中的 Task、Checkpoint、验收标准和验证命令逐项推进。
